## Part A: Project Overview & Architecture

### Purpose
Claude Code Organizer (CCO) solves the problem of visibility and management of Claude Code's scattered configuration files. Claude Code stores memories, skills, MCP server configs, hooks, plans, rules, commands, agents, sessions, plugins, and config files across `~/.claude/` and per-project directories. CCO discovers all of them, groups them by scope hierarchy, and provides both a visual dashboard and an MCP server interface for viewing, moving, and deleting items between scopes.

### Dual-Mode Design
The tool operates in two modes, selected at startup via CLI flag:

1. **Web Dashboard mode** (default): `node bin/cli.mjs` starts an HTTP server (default port 3847) serving a three-panel browser UI. Humans interact via drag-and-drop, search, filter, and button clicks.
2. **MCP Server mode**: `node bin/cli.mjs --mcp` starts an MCP server over stdio transport. AI clients (Claude Code, Cursor, Windsurf) connect and call tools programmatically.

Entry point: `bin/cli.mjs` (lines 1-114). The `--mcp` flag routes to `src/mcp-server.mjs`, otherwise to `src/server.mjs`.

### Zero-Build Philosophy
- Pure ESM (`"type": "module"` in package.json)
- No transpilation, no bundler, no build step
- `src/ui/` files (HTML, CSS, JS) served directly by the Node HTTP server
- CDN-loaded client-side dependencies (SortableJS, Google Fonts)
- Single production dependency: `@modelcontextprotocol/sdk`

### File Structure Overview
```
bin/
  cli.mjs              — Entry point, mode routing, pre-flight checks, /cco skill auto-install
src/
  scanner.mjs          — Pure data module: scope discovery + 11-category item scanner
  mover.mjs            — Pure data module: move + delete items between scopes
  server.mjs           — HTTP server, REST API routes, context budget calculator
  mcp-server.mjs       — MCP server wrapping scanner/mover as 4 tools
  tokenizer.mjs        — Token counting (ai-tokenizer or bytes/4 fallback)
  ui/
    index.html         — Three-panel layout, modals, toast

    style.css          — Full styling with light/dark themes (oklch colors)
.claude-plugin/
  plugin.json          — Plugin metadata for Claude Code plugin registry
  settings.json        — Auto-configures MCP server + bash permission
  skills/organize.md   — Plugin-bundled skill definition
skills/
  organize/SKILL.md    — Standalone skill definition (auto-installed to ~/.claude/skills/cco/)
```

## Part B: Scope Discovery Engine (scanner.mjs)

### What is a "Scope"?
A scope represents a level in Claude Code's configuration hierarchy. There are three types:

| Type | Meaning | Example |
|------|---------|---------|
| `global` | Applies everywhere, stored in `~/.claude/` | Always exists, singleton |
| `workspace` | A parent directory that contains child projects | `/home/user/mycompany` |
| `project` | A specific repo/project directory | `/home/user/mycompany/repo1` |

Each scope has: `id`, `name`, `type`, `tag`, `parentId`, `claudeProjectDir`, `repoDir`.

### How `~/.claude/projects/` Directory Names Encode Filesystem Paths
Claude Code creates subdirectories under `~/.claude/projects/` with names that encode the absolute filesystem path of a project. The encoding replaces `/` with `-` and prepends `-`.

Example: `/home/user/mycompany/repo1` becomes `-home-user-mycompany-repo1`

### The Path Decoding Algorithm: `resolveEncodedProjectPath()` (lines 159-191)

The challenge: directory names themselves may contain dashes (e.g., `my-company`), so the encoding is ambiguous. The algorithm uses **greedy matching** to resolve this:

```javascript
async function resolveEncodedProjectPath(encoded) {
  // Remove leading dash, split by dash
  const segments = encoded.replace(/^-/, "").split("-");
  let currentPath = "/";
  let i = 0;

  while (i < segments.length) {
    // Try longest match first: join remaining segments and check if directory exists
    let matched = false;
    for (let end = segments.length; end > i; end--) {
      const candidate = segments.slice(i, end).join("-");
      const testPath = join(currentPath, candidate);
      if (await exists(testPath)) {
        const s = await safeStat(testPath);
        if (s && s.isDirectory()) {
          currentPath = testPath;
          i = end;
          matched = true;
          break;
        }
      }
    }
    if (!matched) {
      // Try single segment
      currentPath = join(currentPath, segments[i]);
      i++;
    }
  }

  if (await exists(currentPath)) return currentPath;
  return null;
}
```

**Walkthrough example:**
Encoded name: `-home-user-my-company-repo1`
Segments after split: `["home", "user", "my", "company", "repo1"]`

1. Start at `/`, i=0
2. Try longest: `home-user-my-company-repo1` at `/` -- doesn't exist
3. Try `home-user-my-company` -- doesn't exist
4. ... eventually `home` exists at `/home`, match. i=1
5. Try `user-my-company-repo1` at `/home` -- doesn't exist
6. ... `user` exists at `/home/user`, match. i=2
7. Try `my-company-repo1` at `/home/user` -- doesn't exist
8. Try `my-company` at `/home/user` -- exists! match. i=4
9. Try `repo1` at `/home/user/my-company` -- exists! match. i=5
10. Result: `/home/user/my-company/repo1`

### How Parent-Child Relationships Are Determined (lines 257-280)
After resolving all encoded paths, scopes are sorted by path depth (shorter = parent). Then for each scope, the algorithm finds the deepest existing scope whose `realPath` is a prefix of the current scope's path. That becomes the `parentId`.

### How "Workspace" Detection Works (lines 267-269)
A scope is classified as `workspace` (rather than `project`) if:
1. Its parent is `global` (i.e., it's a top-level directory)
2. There exists at least one other project entry whose path starts with this scope's path + "/"

In other words, a workspace is any top-level directory that has child projects under it.

### `discoverScopes()` Function (lines 198-283)
1. Creates the singleton `global` scope
2. Reads all directories in `~/.claude/projects/`
3. For each directory, calls `resolveEncodedProjectPath()` to find the real path
4. Checks if the directory has any content (not just `.DS_Store`)
5. Sorts by path depth, then alphabetically
6. Builds parent-child relationships
7. Classifies each as workspace or project

## Part C: The 11-Category Scanner (scanner.mjs)

All scanners are called in parallel per scope (line 986-997). Plugins are scanned separately (global only, line 1002).

### 1. Memories -- `scanMemories()` (lines 325-364)

**Where it looks:**
- Global: `~/.claude/memory/*.md` (excluding `MEMORY.md`)
- Project: `~/.claude/projects/<encoded>/memory/*.md` (or custom dir from `settings.autoMemoryDirectory`)

**What it parses:** YAML frontmatter via `parseFrontmatter()` (lines 122-132). Extracts `name`, `description`, `type` (feedback, user, project, reference).

**Fields extracted:** `category:"memory"`, `scopeId`, `name` (from frontmatter or filename), `fileName`, `description`, `subType` (from frontmatter `type`), `size`, `sizeBytes`, `mtime`, `ctime`, `path`.

### 2. Skills -- `scanSkills()` (lines 366-452)

**Where it looks:**
- Global: `~/.claude/skills/` + managed dir (`/Library/Application Support/ClaudeCode/.claude/skills/` on macOS, `/etc/claude-code/.claude/skills/` on Linux)
- Project: `<repoDir>/.claude/skills/` (skipped if repoDir === HOME via `isGlobalClaudeDir()`)

**What it parses:** Reads `SKILL.md` in each subdirectory. Extracts description from first meaningful paragraph line after the heading (not frontmatter). Also loads **skill bundle info** from `skills-lock.json` files via `loadSkillBundles()` (lines 295-321).

**Special logic:**
- Supports both real directories and symlinks (line 389)
- Skips "private" directory (line 391)
- Counts files in skill directory and calculates total size
- Bundle detection: checks `<repoDir>/skills-lock.json` (v1) and `~/.agents/.skill-lock.json` (v3)

**Fields extracted:** `category:"skill"`, `name` (directory name), `description`, `size` (total dir size), `fileCount`, `bundle` (source URL from lock file), `path` (directory path).

### 3. MCP Servers -- `scanMcpServers()` (lines 454-598)

**Where it looks (multi-source scanning):**
For global scope, scans ALL these sources to show duplicates:
1. `~/.claude/.mcp.json` (user scope via `claude mcp add -s user`)
2. `~/.mcp.json` (alternate user location)
3. `~/.claude.json` top-level `mcpServers` (user scope via `claude mcp add` default)
4. `/etc/claude-code/managed-mcp.json` (enterprise managed)
5. `mcpServers` inside `settings.json` / `settings.local.json`

For project scope:
1. `<repoDir>/.mcp.json`
2. `~/.claude.json` → `projects[repoDir].mcpServers`
3. `<repoDir>/.claude/settings.json` / `settings.local.json` mcpServers

**What it parses:** JSON files. Iterates `mcpServers` object entries.

**Fields extracted:** `category:"mcp"`, `name` (server name key), `description` (command + args joined), `mcpConfig` (full server config object), `path` (which JSON file it came from).

### 4. Configs -- `scanConfigs()` (lines 601-640)

**Where it looks:**
- Global: `~/.claude/CLAUDE.md`, `~/.claude/settings.json`, `~/.claude/settings.local.json`, managed `CLAUDE.md`, `managed-settings.json`
- Project: `<repoDir>/CLAUDE.md`, `<repoDir>/.claude/CLAUDE.md`, `<repoDir>/.claude/settings.json`, `<repoDir>/.claude/settings.local.json`

**What it parses:** Only checks existence and stats. Does not parse content.

**Special:** All config items have `locked: true` -- they cannot be moved or deleted through the UI API (but the UI generates Claude Code prompts for managing them).

### 5. Hooks -- `scanHooks()` (lines 642-689)

**Where it looks:**
- Global: `settings.json`, `settings.local.json`, `managed-settings.json`
- Project: `<repoDir>/.claude/settings.json`, `<repoDir>/.claude/settings.local.json`

**What it parses:** JSON. Reads `settings.hooks` object. Each hook event has an array of hook groups, each group has a `hooks` array of individual commands.

**Fields extracted:** `category:"hook"`, `name` (event name like "PreToolUse"), `description` (command or prompt text), `subType` (command type). All hooks are `locked: true`.

### 6. Plugins -- `scanPlugins()` (lines 691-726)

**Where it looks:** `~/.claude/plugins/cache/<org>/<plugin>/` -- global only.

**Special logic:** Skips `temp_*` directories and hidden dirs. Only scans at plugin name level (not version dirs).

**Fields extracted:** `category:"plugin"`, `name` (plugin name), `fileName` (org/plugin), `locked: true`.

### 7. Plans -- `scanPlans()` (lines 728-773)

**Where it looks:**
- Global: `~/.claude/plans/` (or custom dir from `settings.plansDirectory`)
- Project: `~/.claude/projects/<encoded>/plans/`

**What it parses:** Markdown files. Extracts first `# heading` as description.

### 8. Rules -- `scanRules()` (lines 775-821)

**Where it looks:**
- Global: `~/.claude/rules/*.md`
- Project: `<repoDir>/.claude/rules/*.md`

**What it parses:** Markdown files. Extracts first `# heading` as description.

### 9. Commands -- `scanCommands()` (lines 823-861)

**Where it looks:**
- Global: `~/.claude/commands/*.md`
- Project: `<repoDir>/.claude/commands/*.md`

**What it parses:** YAML frontmatter for `name` and `description`.

### 10. Agents -- `scanAgents()` (lines 863-901)

**Where it looks:**
- Global: `~/.claude/agents/*.md`
- Project: `<repoDir>/.claude/agents/*.md`

**What it parses:** YAML frontmatter for `name` and `description`.

### 11. Sessions -- `scanSessions()` (lines 903-967)

**Where it looks:** `~/.claude/projects/<encoded>/*.jsonl` -- project scopes only (line 904: returns empty for global).

**What it parses:** JSONL files. Uses **chunked I/O** for performance:
- `readFirstLines(path, 10)` (lines 54-78): reads first 10 lines via 8KB chunks to find `aiTitle` near the top (line 4-6)
- `readLastLines(path, 30, fileSize)` (lines 80-110): reads last 30 lines via reverse 8KB chunks to find the last user message

**Special logic:**
- Title extraction: searches head lines for `aiTitle` field in JSONL
- Description: scans tail lines in reverse for last user message with role "user", skips XML-like and tool_use content
- Sessions have `deletable: true` but are NOT movable

### Category-to-Path Mapping

| Category | Global Path | Project Path | Resolution Function |
|----------|-------------|--------------|---------------------|
| memory | `~/.claude/memory/` | `~/.claude/projects/<id>/memory/` | `resolveMemoryDir()` (line 40-43) |
| skill | `~/.claude/skills/` | `<repoDir>/.claude/skills/` | `resolveSkillDir()` (line 45-50) |
| plan | `~/.claude/plans/` | `~/.claude/projects/<id>/plans/` | `resolvePlanDir()` (line 52-55) |
| rule | `~/.claude/rules/` | `<repoDir>/.claude/rules/` | `resolveRuleDir()` (line 57-62) |
| command | `~/.claude/commands/` | `<repoDir>/.claude/commands/` | `resolveCommandDir()` (line 64-69) |
| agent | `~/.claude/agents/` | `<repoDir>/.claude/agents/` | `resolveAgentDir()` (line 71-76) |
| mcp | `~/.claude/.mcp.json` | `<repoDir>/.mcp.json` | `resolveMcpJson()` (line 78-83) |

### `validateMove()` (lines 87-105)
Checks and blocks:
1. `item.locked` -- locked items cannot move
2. Same scope (no-op) -- returns error
3. Category must be in `["memory", "skill", "mcp", "plan", "command", "agent", "rule"]`

### How MCP Server Moves Work (lines 249-300)
MCP servers are entries inside JSON files, not standalone files. The move:
1. Read source `.mcp.json`, extract the server config by name
2. Read (or create) destination `.mcp.json`
3. Check destination doesn't already have a server with that name
4. Add to destination's `mcpServers` object
5. Delete from source's `mcpServers` object
6. Write both files

### `safeRename()` and EXDEV Fallback (lines 21-33)
Wraps `fs.rename()`. If it throws `EXDEV` (cross-device link -- source and dest are on different filesystems/mount points), falls back to `cp()` + `rm()`.

### `getValidDestinations()` Filtering Logic (lines 418-444)
Returns scopes where the item can move to:
- Excludes current scope
- `memory`, `mcp`, `plan`: can go to **any** scope
- `skill`, `command`, `agent`, `rule`: can go to `global` or any scope with a `repoDir`
- Locked items: returns empty array

### Delete Functions (lines 340-412)
- `deleteMemory/Plan/Command/Agent/Rule`: calls `unlink()` on the file
- `deleteSkill`: calls `rm(path, { recursive: true })` -- deletes entire skill directory
- `deleteMcp`: reads JSON, deletes the server key, rewrites the file
- `deleteSession`: deletes `.jsonl` file + any subagent directory with the same UUID

## Part E: Context Budget Calculator (server.mjs)

### What "Context Budget" Means
The context budget is the number of tokens consumed by Claude Code's system setup BEFORE the user types anything. It represents the overhead of having customizations (CLAUDE.md files, skills, rules, MCP configs, etc.) loaded into the conversation context.

### Route: `GET /api/context-budget?scope=<id>&limit=<number>` (lines 136-356)

### Always-Loaded Items (lines 156-176)
These are in context every request:
- **System prompt**: ~6,500 tokens (estimated constant)
- **System tools (loaded part)**: ~6,000 tokens (estimated constant)
- **CLAUDE.md files**: full content, from all ancestor directories
- **Rules** (`.claude/rules/*.md`): unconditional, no paths frontmatter filtering
- **MEMORY.md**: first 200 lines or 25KB (line 267-268)
- **Skill descriptions**: name + frontmatter description
- **Commands and Agents**: full file content

Categories always loaded: `skill`, `rule`, `command`, `agent` (line 176).
Config items loaded: only `CLAUDE.md`, `.claude/CLAUDE.md`, `CLAUDE.md (managed)` (line 178).

### Deferred Items (lines 165-173)
Reserved but loaded on-demand via ToolSearch:
- **MCP tool definitions**: ~3,100 tokens per unique server (line 308). Claude deduplicates by name (priority: local > project > user), so unique names are counted, not total entries.
- **System tools (deferred part)**: ~10,500 tokens

### How Scope Chain Walking Works (lines 146-153)
The budget calculator collects the parent chain by walking up `parentId` links. Items from parent scopes are classified as "inherited". The API tokenizes current-scope items and inherited items separately, then sums them.

### ai-tokenizer Integration and Fallback (tokenizer.mjs, lines 1-59)
- Lazy initialization on first use (line 19-38)
- Tries to import `ai-tokenizer` + `ai-tokenizer/encoding` with Claude encoding
- If not installed, falls back to `Buffer.byteLength(text, "utf-8") / 4` (~75-85% accurate)
- Returns `{ tokens, confidence: "measured"|"estimated" }`

### System Overhead Constants (lines 293-296)
```javascript
const SYSTEM_LOADED = 12500;   // system prompt (~6.5K) + system tools loaded (~6K)
const SYSTEM_DEFERRED = 10500; // system tools deferred
// MCP: mcpUniqueCount * 3100 tokens per unique server
```

### Response Structure
Returns JSON with:
- `alwaysLoaded`: `{ currentScope, inherited, system, total }`
- `deferred`: `{ currentScope, inherited, systemTools, mcpToolSchemas, mcpServerCount, mcpUniqueCount, total }`
- `total`, `contextLimit` (default 200K), `percentUsed`, `percentWithDeferred`
- `method` ("measured" or "estimated")

### Three-Panel Layout Structure (index.html)

1. **Left Sidebar** (`#sidebar`): Scope tree with expandable nodes showing category counts. Contains search input, collapse-all button, theme toggle, export button, footer links.
2. **Main Content** (`#mainContent`): Content header with scope name, inheritance chain, context budget button. Filter bar with category pills. Item list grouped by category with sort controls.
3. **Right Detail Panel** (`#detailPanel`): Item metadata (scope, type, description, size, dates, path). File preview. "Claude Code Prompt" action buttons. Move/Delete/Open buttons. Also hosts the **Context Budget Panel** (`#ctxBudgetPanel`) as an alternative right panel.

Panels are separated by draggable **resizers** (`#resizerLeft`, `#resizerRight`) implemented in `setupResizer()` (app.js lines 397-438).

### How Drag-and-Drop Works (SortableJS Integration)

**Library:** SortableJS v1.15.6 loaded from CDN (index.html line 9).

**Initialization:** `initSortable()` (app.js lines 1310-1378) creates Sortable instances on all `.sortable-zone` elements. Each zone has a `data-group` attribute matching the category group (memory, skill, mcp, etc.). Items with the same group can be dragged between zones.

**Flow:**
1. `onStart`: Stores the dragged item, adds `drag-active` class to sidebar, collapses sidebar to scope-tree-only view
2. User drags item to a different scope's sortable zone OR drops on a sidebar scope block
3. `onEnd`: If source != destination zone, shows the drag confirm modal (`showDragConfirm()`)
4. User confirms -> `doMove()` calls `POST /api/move`
5. After move, `refreshUI()` re-scans and re-renders everything

**Sidebar drop zones** (lines 1380-1444): Separate `dragover`/`drop` handlers on document allow dropping items onto sidebar scope blocks (not just sortable zones). Adds visual `.drop-target` highlight.

**Locked items on drag**: If a locked item is dragged to a scope block, instead of moving via API, it generates a Claude Code prompt and copies it to clipboard (lines 1413-1421).

### Search and Filter Implementation

**Search** (app.js lines 145-151): Filters items by matching search query against `[name, description, category, subType, path].join(" ")` (line 2143-2153). Also filters sidebar scope visibility.

**Category Filters** (app.js lines 196-221): Toggle filter pills. `activeFilters` Set stores active categories. Items must match at least one active filter (or all if none active).

### Undo System

**Move undo** (app.js lines 1835-1858): After a successful move, creates an `undoFn` that calls `POST /api/move` in reverse (moves the item back). Shows toast with "Undo" button for 8 seconds.

**Delete undo** (app.js lines 1869-1952):
1. Before deleting, reads the file content as backup (`backupContent`)
2. For MCP items, stores `{ name, config, mcpJsonPath }` as `mcpBackup`
3. After successful delete, creates an `undoFn` that calls either `POST /api/restore` or `POST /api/restore-mcp`
4. For skills (directories), restores by creating the directory and writing SKILL.md

**Toast** (app.js lines 1961-1983): Shows notifications at bottom. Supports error styling, undo button, persistent (close button), and auto-hide (4s default, 8s with undo).

### Additional UI Features

**Bulk operations** (lines 311-357): Checkbox select mode. Bulk move (same category only) and bulk delete.

**Claude Code Prompt buttons** (`renderCcActions()`, lines 811-896): Category-specific prompts that users can copy and paste into Claude Code. Includes "Explain This", "Edit", "Resume Session", "Modify", "Remove" etc. Each prompt is multi-step and asks Claude to read, explain, and confirm before acting.

**Context Budget panel** (lines 941-1143): Visual bar chart showing token usage. Supports sort by scope/category/tokens. Toggleable 200K/1M context window. Collapsible sections.

**Theme toggle**: Toggles `.dark` class on body (line 364). CSS uses `oklch` color space with separate light/dark variable sets.

**Export** (lines 913-939): Calls `POST /api/export`, which copies all items to `~/.claude/exports/cco-backup-<timestamp>/` organized by `scopeId/category/`.

## Part G: MCP Server Mode (mcp-server.mjs)

### Overview
File: `src/mcp-server.mjs` (135 lines). Wraps `scanner.mjs` and `mover.mjs` functions as 4 MCP tools using `@modelcontextprotocol/sdk`.

### Transport
Stdio (`StdioServerTransport` from the SDK, line 133-134). AI clients connect via stdin/stdout.

### The 4 Tools

#### 1. `scan_inventory` (lines 41-51)
- **Parameters:** None
- **Returns:** Full scan result as JSON (scopes, items, counts)
- **Calls:** `freshScan()` which calls `scan()` from scanner.mjs

#### 2. `move_item` (lines 53-79)
- **Parameters:**
  - `category`: enum of `['memory', 'skill', 'mcp', 'plan', 'session', 'command', 'agent', 'rule']`
  - `name`: string
  - `fromScopeId`: string
  - `toScopeId`: string
- **Logic:** Calls `freshScan()` if no cached data, finds item via `findItem()`, calls `moveItem()` from mover.mjs
- **After success:** Refreshes cache via `freshScan()`

#### 3. `delete_item` (lines 81-106)
- **Parameters:** `category`, `name`, `scopeId`
- **Logic:** Same pattern as move_item. Calls `deleteItem()` from mover.mjs.

#### 4. `list_destinations` (lines 108-131)
- **Parameters:** `category`, `name`, `scopeId`
- **Returns:** Array of valid destination scopes
- **Calls:** `getValidDestinations()` from mover.mjs

### Item Lookup: `findItem()` (lines 32-39)
Searches cached scan data by `category`, `name` (or `fileName`), and `scopeId`.

### Caching
`cachedData` variable (line 21) stores scan results. Refreshed on `scan_inventory` and after successful move/delete.

## Part H: Plugin & Skill System

### `.claude-plugin/` Structure

This directory makes CCO installable as a Claude Code plugin:

1. **`plugin.json`**: Plugin metadata (name, version, description, author, homepage, keywords).
2. **`settings.json`**: Auto-configures when the plugin is installed:
   - Adds `claude-code-organizer` MCP server (via npx)
   - Adds bash permission for `Bash(npx @mcpware/claude-code-organizer*)`
3. **`skills/organize.md`**: Plugin-bundled skill with frontmatter (`name: organize`, `description`, `argument-hint: "[port]"`).

### How Auto-Install of `/cco` Skill Works (cli.mjs, lines 36-61)

When the dashboard starts (non-MCP mode):
1. Check if `~/.claude/skills/cco/SKILL.md` exists
2. If not, create the directory and write SKILL.md with:
   ```yaml
   ---
   name: cco
   description: Open Claude Code Organizer dashboard to manage memories, skills, MCP servers across scopes
   ---
   ```
   Plus body text explaining how to run `npx @mcpware/claude-code-organizer`.
3. Print success message: "Installed /cco skill globally"
4. Silently fails if write permissions are missing

### Skill Definition Format

**Standalone skill** (`skills/organize/SKILL.md`):
```yaml
---
name: organize
description: Open the Claude Code Organizer dashboard...
argument-hint: [--port <number>]
allowed-tools: [Bash, Read]
---
```

## Part I: Technologies & Dependencies

### Runtime Requirements
- Node.js >= 20 (`"engines": { "node": ">=20" }` in package.json)
- No OS restriction (works on macOS, Linux, Windows)

### Production Dependencies
- **`@modelcontextprotocol/sdk` ^1.27.1**: MCP server framework (McpServer, StdioServerTransport, zod for schema validation)

### Optional Dependency
- **`ai-tokenizer` ^1.0.6**: Claude-specific tokenizer for accurate token counting. If not installed, falls back to bytes/4 estimation.

### Dev Dependency
- **`@playwright/test` ^1.58.2**: End-to-end browser testing

### CDN-Loaded Client Libraries
- **SortableJS 1.15.6**: `https://cdn.jsdelivr.net/npm/sortablejs@1.15.6/Sortable.min.js` -- drag-and-drop library
- **Google Fonts**: Lato (400, 700, 900) and Geist Mono (400, 500)

### CI/CD Pipeline (`.github/workflows/publish.yml`)

Triggered on push of `v*` tags. Single job with these steps:

1. **Checkout** (actions/checkout@v5)
2. **Setup Node.js** (LTS, npm registry)
3. **Install dependencies** (`npm ci` or `npm install` with `--ignore-scripts`)
4. **Build** (runs `npm run build` if script exists -- it doesn't, so no-op)
5. **Dedup check**: Queries npm registry to see if version already published
6. **Publish to npm**: `npm publish --access public` with `NPM_TOKEN` secret. Uses npm provenance (`"publishConfig": { "provenance": true }` in package.json)
7. **Create GitHub Release**: via `softprops/action-gh-release@v2` with auto-generated release notes
8. **Install mcp-publisher**: Downloads the MCP Registry publisher binary
9. **OIDC auth**: Authenticates to MCP Registry via GitHub Actions OIDC
10. **Publish to MCP Registry**: Registers/updates the MCP server entry (continues on error)

### Permissions Required
- `id-token: write` -- for MCP Registry OIDC authentication
- `contents: write` -- for creating GitHub Releases

### Platform-Aware Paths
- macOS managed dir: `/Library/Application Support/ClaudeCode/`
- Linux managed dir: `/etc/claude-code/`
- Set in scanner.mjs lines 17-20

### HTTP Server Details
- Built on Node's built-in `http.createServer` (no Express/Koa)
- Default port: 3847
- Auto-retry: tries up to 10 consecutive ports if default is in use (server.mjs lines 650-687)
- Auto-opens browser on macOS (`open`) and Linux (`xdg-open`) -- cli.mjs lines 108-113
- Non-blocking update check on startup (queries npm registry)
